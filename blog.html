Joining the Center for Game Science
1/4/16 2:40 PM
December saw a lot of changes for my plans, the most significant being my decision to work the UW's Center for Game Science. This will be an awesome opportunity for me to work with game developers and gain semi-professional experience. The game I will be working on is Nanocrafter, which is a puzzle game based off of synthetic biology (specifically, how DNA strands interact with each other). This is very exciting to me because it's a pretty fun way to learn about synthetic biology. Seriously, try playing it, I thought it was fun. The current push is to refactor the code and improve some of the games features.

This means my other ideas will be on hold. I currently have two other projects on deck: A JavaScript RPG game, and a electrical circuit sandbox game. I will complete these over the summer. The Brick Pong team lost a lot of steam last quarter, so it looks like a dead project. Later on I might consider going back and finishing it.

Anyway, Nanocrafter was made in ActionScript 3.0, which seems like an old programming language. The last time it was updated was in 2006. Yet, ActionScript seems to be pretty powerful and a pretty good choice for game development. Every software developer and game developer I've talked to didn't even mention ActionScript when I asked them what languages are useful in today's industry. Regardless, I took some time over break to make a pong game with ActionScript to get myself familiar with it. It was actually a lot of fun. I'm excited to work with the Center for Game Science this quarter.
RPG Maker MV
12/1/15 11:50 PM
On my page for The Shadow Mountain I mentioned how I felt RPG Maker VX had a limited scope. It did. It still does. But Enterbrain's latest release, RPG Maker MV, is orders of magnitude better (an intermediate release, RPG Maker VX Ace, was pretty good too, but MV is even better). I played around with the trial version last night, and it improved tons of things I hated in RPG Maker VX. I was so impressed that I decided to share my thoughts on it to my blog.

RPG Maker MV exports to iOS, Android, WebGL, OSX, and Windows. An incredible improvement from RPG Maker VX's only export option being Windows. It also uses JavaScript instead of Ruby now. I'm not quite proficient in either of those languages, but from what I saw at Dubhacks, JavaScript is quite popular, with things like angular and node.js. xkcd even made a cartoon suggesting that in 2020 we'll have operating systems ending in .js. If xkcd counts as a credible source, then Enterbrain is right there on the cutting edge of technology.

The fact it can make games playable in-browser on a mobile device is really impressive. That's something Unity can't do very well. (Unity can export to WebGL but it's still in beta and builds don't quite work as expected) In case the implications of that aren't clear, RPG Maker MV had an advantage over Unity. There are also tons of new tools in the RPG Maker MV editor, making things I had wished possible with RPG Maker VX now simple matters. Easy plug-in support, a character generator, terrain tags, more freedom with iconsets and tilesets, a clearer battle system, more flexible passibility settings, and a built in caterpillar party system are just some of the improvements.

I don't think I will ever use RPG Maker MV to seriously make games, though. It's for a very specific game genre, and I need full flexibility for my ideas. Still, well done Enterbrain. I was thoroughly impressed with MV. Maybe once I'm past the years of frantically maintaining a high enough grade point average to get into UW's computer science program I'll set aside some time and make an RPG Maker game for fun.
Eighteen out of Twenty
11/28/15 12:02 AM
My computer science professor used the phrase "eighteen out of twenty" to describe why Java's syntax for Array length and String length is different (.length vs. .length()). The developers of Java made a small mistake; two groups had created different conventions. But they were tired, and getting an 18/20 on the "assignment" of making Java was good enough. I think my professor was trying to describe it in terms that a college student would understand: the makers of Java felt it was good enough and were too tired to put in the extra effort for a mere two points.

Personally, I agonize over missing 2 points on an assignment. But honestly, with this website, I'd have to say I'm okay with 18/20. It's actually probably closer to 14/20, because of internal issues that I will explain further down. Now if I was a successful, rich, popular developer, I would aim for 20/20. But perfection isn't important just yet because this is a learning experience more than anything else. Rather than try to get everything perfect the first time, I'm going to focus on improving my abilities, which in turn will organically invoke improvement on my actual projects.

So what needs to be improved? Take a look at the source files for this site (the CSS file too). They are unorganized, cluttered, redundant, and inconsistent. AND, my blogging system is just adding more text to the blog.html webpage. This blog post was literally typed into the HTML file. It's not terrible I suppose, but what if I changed the template? What if I redesigned the HTML containing the blog posts? I would have to go back through each post and change it. Same with the navigation bar, the header, the footer, and the meta data on each page. Typing everything by hand only works until you need to change something.

The solution is to write a program that takes an HTML template file and several text source files, and generates the website. I think this is what Jekyll does. I could learn how to use Jekyll, but it will be more fun and more educational if I write the program myself. Rather than limiting myself to knowing how to generate websites with a specific program, I can unravel the concepts myself, ensuring that I will be able to do the same thing 10 years from now when Jekyll is outdated.

Tomorrow I need to return to homework, research and studying, so for now, typing the blogs directly into the HTML file is going to have to be acceptable.
